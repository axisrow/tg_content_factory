# Code Review: TG Post Search / Content Factory

Вот подробный обзор (Code Review) проекта `tg_content_factory`.

### 📊 Общее резюме
Проект представляет собой **хорошо спроектированную, модульную систему** для мониторинга Telegram-каналов. Архитектура четко разделена на слои, используется современный асинхронный стек (FastAPI, aiosqlite, Telethon). Код чистый, типизированный и следует стандартам PEP 8.

Однако обнаружено **критическое расхождение** между документацией и реализацией в плане безопасности (хранение сессий), а также отсутствуют некоторые стандартные защиты для веб-приложений (CSRF).

---

### 1. Архитектура и Дизайн
**Сильные стороны:**
*   **Слоистая архитектура:** Четкое разделение ответственности:
    *   `src/web/`: Веб-интерфейс и API.
    *   `src/telegram/`: Взаимодействие с Telegram (ClientPool, Collector).
    *   `src/search/`: Логика поиска (локальная БД + LLM).
    *   `src/database.py`: Слой доступа к данным.
*   **Асинхронность:** Весь проект (от БД до сетевых вызовов) построен на `asyncio`, что идеально для I/O-bound задач (скрейпинг, веб-сервер).
*   **Инъекция зависимостей:** Компоненты (`Database`, `ClientPool`) передаются в конструкторы сервисов, что упрощает тестирование.

**Зоны роста:**
*   **`src/main.py` перегружен:** CLI-команды содержат много бизнес-логики, которую лучше вынести в сервисные классы.
*   **Миграции БД:** Метод `_migrate` в `database.py` делает ручные проверки через `PRAGMA`. Это хрупкий подход. Для развития проекта лучше внедрить `Alembic` или аналогичный инструмент.

### 2. Качество кода
**Сильные стороны:**
*   **Type Hinting:** Активное использование аннотаций типов (`list[int]`, `dict | None`) помогает в поддержке и отлове ошибок.
*   **Pydantic:** Используется для валидации конфигов и данных, что надежнее ручных проверок.
*   **Конфигурация:** Отличная реализация `config.yaml` с подстановкой переменных окружения (`${ENV_VAR}`).

### 3. Безопасность 🛡️
**Критические замечания:**
*   **Хранение сессий (Finding):** В `CLAUDE.md` заявлено: *"Session strings stored encrypted in DB"*. Однако в `src/database.py` (метод `add_account`) строка сессии сохраняется как **обычный текст**:
    ```python
    VALUES (?, ?, ?, ?, ?) -- session_string передается как есть
    ```
    Если база данных утечет, злоумышленник получит полный доступ к подключенным аккаунтам.
*   **CSRF Protection:** В веб-интерфейсе не обнаружено защиты от CSRF (Cross-Site Request Forgery). Если администратор зайдет на вредоносный сайт, злоумышленник может отправить скрытый запрос на `/channels/delete` или `/scheduler/stop`.

**Хорошие практики:**
*   **Auth Middleware:** Простая, но эффективная реализация Basic Auth и cookie-сессий для однопользовательского режима.
*   **Секреты:** API ключи и пароли загружаются через переменные окружения.

### 4. Производительность и Надежность
**Сильные стороны:**
*   **СУБД:** Использование `aiosqlite` и `FTS5` (Full-Text Search) обеспечивает быстрый поиск по текстам сообщений без внешних зависимостей типа Elasticsearch.
*   **Batch Operations:** Сообщения вставляются пачками (`executemany`), что снижает нагрузку на диск.
*   **Flood Wait:** `ClientPool` грамотно обрабатывает ошибки `FloodWaitError` от Telegram, временно исключая "уставшие" аккаунты из ротации. Это критически важно для стабильной работы скрейпера.

### 5. Тестирование
**Сильные стороны:**
*   **Интеграционные тесты:** `tests/test_integration.py` покрывает полный цикл работы: от авторизации до создания каналов и поиска.
*   **Mocking:** Грамотное использование `unittest.mock` для эмуляции ответов Telegram API.

---

### Рекомендации по улучшению

1.  **Исправить безопасность хранения сессий:** Реализовать шифрование `session_string` перед записью в БД (используя `cryptography.fernet`), как и было задумано в документации. Ключ шифрования можно хранить в ENV или генерировать из `WEB_PASS`.
2.  **Добавить CSRF-защиту:** Внедрить middleware (например, `starlette-csrf`) для защиты форм в админке.
3.  **Рефакторинг `main.py`:** Вынести логику команд (особенно `collect` и `search`) в отдельные контроллеры/сервисы.
4.  **Улучшить миграции:** Если планируется развитие схемы БД, перейти на `Alembic`.
